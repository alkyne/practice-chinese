취약점은 작은 버그에서 시작된다

아주 작은 코딩 실수가 심각한 보안 취약점이 됩니다.

// 버퍼 오버플로우 취약점 예시
char buf[1024];
scanf("%s", buf);  // 입력 길이 제한 없음 → 버퍼 오버플로우!
  • 보안 연구원들이 취약점을 발견한다
  • 개발사가 패치를 배포한다
  • 또 다른 취약점이 발견된다
  • 이 사이클이 십수 년째 반복 중
  • 방어 기법도 계속 현대적으로 진화

🔬 대학원

탐구하는 과정

대학원은 수업을 더 듣는 곳이 아니라,
아직 아무도 완전히 풀지 못한 문제를 다루는 곳입니다.
→ 그것이 바로 연구입니다.

대학원에서 듣는 수업 = 풀리지 않은 문제를 풀기 위한 기초 지식을 쌓는 과정

연구란 무엇인가

연구는 이런 질문에서 시작됩니다

  • 왜 이 방법은 잘 안 될까?
  • 더 빠르게 할 수 없을까?
  • 더 안전하게 만들 수 없을까?
  • 더 적은 데이터로 더 잘할 수 없을까?

연구의 기본 구조

1. 어떤 문제가 있는지 발견한다
2. 기존 방법이 왜 부족한지 본다
3. 새로운 방법을 제안한다 (핵심!)
4. 실험해서 나아졌는지 확인한다

연구 예시 ① 이미지 분류 성능 향상

고양이와 강아지를 구분하는 문제

예전 "귀가 뾰족하면 고양이?" 같은 규칙을 사람이 직접 설정
풀린 문제 고양이/강아지 같은 단순 객체 분류
어려운 문제 조명, 가림, 해상도 저하 시 정확도 하락
안 풀린 문제 컴퓨터는 진정 "고양이"라는 개념을 이해한 것인가?
사람처럼 적은 예시만 보고 일반화할 수 있는가?
세상에 없던 새로운 것도 분류할 수 있는가?

연구 예시 ② 대규모 웹 서비스

서버가 자꾸 죽는다!

예전 서버 1대가 죽으면 서비스 전체가 멈춤
풀린 문제 캐싱 / 로드밸런싱 / 컨테이너 & 오케스트레이션 / 모니터링 / CDN
어려운 문제 마이크로서비스 장애 원인 추적 / 부분→전체 장애 전파
배포 중 장애 / "서버는 살아있는데 서비스는 느린" 애매한 상태
안 풀린 문제 완전히 자동으로 장애를 예측하고 복구하는 것

연구 예시 ③ 소프트웨어 해킹 연구

예전 취약한 코드가 있으면 바로 공격 가능
풀린 문제 메모리 오류 탐지, 공격 기법 및 취약점 유형의 이해, 방어 패턴 정립
어려운 문제 "버그 탐지"와 "실제 exploit 가능성 판단" 사이의 간극
소스코드 없는 환경(binary only)에서 LLM을 활용한 버그 찾기
안 풀린 문제 소프트웨어를 만들 때부터 취약점이 안 생기게 하는 것

LLM × 해킹 연구 — 프로그램 언어(소스코드)도 결국 언어이기 때문에 LLM 활용 연구가 크게 확장 중
